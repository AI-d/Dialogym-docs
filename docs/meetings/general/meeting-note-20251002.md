## 3. 주요 논의

### 3-1. 발화 정확도/속도 측정 기준

* **정확도**: 문자 단위의 CER(Character Error Rate) 사용.  
  * CER = (삽입 + 삭제 + 치환) / 총 문자 수  
  * 실시간 STT 편차를 고려해 세션 종료 후 전사본 기준 재계산(리포트용) 수행.  
* **속도(WPM)**: `WPM ≈ (문자 수 / 5) ÷ 발화 시간(분)`으로 산출.  

* **판정 구간**(합의):
  * 120 미만: “너무 느림”
  * 120~160: “적절”
  * 160~180: “약간 빠름”
  * 180 이상: “너무 빠름”

> 참고: 팀 문서의 점수화 항목(발화 속도·추임새·공손도·명료성)과 WPM 구간 정의를 그대로 채택.

### 3-2. 팀 역할 및 모델 선택

* **A+B 팀**: WebRTC 기반 **GPT-4o Realtime**로 **저지연 대화**(STT/LLM/TTS 통합)  
* **C 팀**: 세션 종료 후 **GPT-4 또는 4o 텍스트, 제미나이** 호출 → 점수화/요약/개선안 생성(**일관성·비용** 관점에서 유리).  

### 3-3. 실시간 파이프라인(요약)
```
브라우저(getUserMedia) → WebRTC(Opus) → Janus
→ (Opus→PCM 16k 변환) → GPT-4o Realtime(WS)
→ AI 음성 응답(PCM) → 사용자 스피커
```
* **시그널링**: WebSocket(Offer/Answer/ICE), STUN/TURN 기본 구성.  
* **오디오**: 16kHz, 100ms 청크 처리 권장.  
* **메트릭 탭**: 문자수, 경과시간, WPM, CER(추정), filler(“음/어/그/저기”) 카운트

### 3-4. 문서/도구 운영 원칙

* **Docs Repo(공식)**: 회의록/ADR/요구사항 등 **확정 문서** 축적.  
* **Wiki(참조)**: API/환경변수/트러블슈팅 등 **자주 바뀌는 개발 레퍼런스**.  
* **Jira(작업 관리)**: 1주 스프린트, `Backlog→To Do→In Progress→Code Review→Done` 흐름, **CR 필수**. 커밋 메시지에 이슈 키 포함.  
* **Discord(소통)**: 공지/일반/daily/backend/frontend/잡담 + `회의실` 음성 채널 구성 원칙 준수.  
* **협업 흐름**: Discord 제안 → Discussions 논의 → Docs 확정 → Jira 발행 → Wiki 참조 업데이트.

### 3-5. 이슈 라벨 체계(요약)

* Area(Type/우선순위/상태/메타) 표준 라벨 세트 사용(예: `area: webrtc`, `feature`, `priority: P1`, `status: in progress`, `help wanted`).

## 4. 결정 사항

* **정확도**는 **CER**로 평가, **속도**는 문자수 기반 **WPM**으로 산정(상기 구간 적용).  
* **A+B=Realtime**, **C=텍스트 모델** 역할 분리를 유지(일관성/비용/장애격리).  
* **WebRTC–Janus–GPT-4o** 아키텍처 1안으로 진행(오디오 변환 및 WS 연동 포함).  
* 문서/작업/소통 도구 운영 원칙은 각 가이드에 따름(Wiki/Jira/Discord/Workflow).  
* 이슈 라벨은 표준 표에 따라 프로젝트 전반에 통일 적용.
