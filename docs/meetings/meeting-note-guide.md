## 핵심 요약 (Executive Summary)

본 문서는 팀 회의록 작성 방법과 원칙을 정의합니다.
회의록은 의제, 논의 내용, 결정 사항, 개별 담당 작업을 명확히 기록하여 팀 전체가 동일한 정보를 공유합니다.
파일명은 `meeting-note-YYYY.MM.DD.md` 형식을 따르며, 회의 당일 또는 익일 내 작성을 완료합니다.

## 문서 개요 (Overview)

본 문서는 팀 회의의 내용을 체계적으로 기록하여 팀원 간 정보 공유와 프로젝트 진행 상황 추적을 용이하게 합니다.
회의록은 단순 기록이 아닌 의사결정의 근거이자 작업 분배의 기준이 되므로, 명확하고 구체적으로 작성해야 합니다.

---

## 파일명 규칙

```
meeting-note-YYYY.MM.DD.md

예시:
meeting-note-2025.10.06.md
meeting-note-2025.10.13.md
```

## 섹션별 작성 가이드

### 1. 오늘 다룬 의제

**목적**: 회의에서 논의할 주제를 사전에 명확히 정리

**작성 방법**:
- 구체적인 명사형으로 작성
- 우선순위 순서로 나열
- 한 줄로 핵심만 표현

**좋은 예**:
```markdown
* Sprint 2 백로그 우선순위 결정
* ERD 설계 최종 검토
* 배포 환경 구축 방법 논의
```

**나쁜 예**:
```markdown
* 이것저것 이야기
* 개발 관련
* 기타
```

---

### 2. 주요 논의

**목적**: 각 팀원의 의견과 제안 사항을 기록

**작성 방법**:
- 팀원별로 구분하여 작성
- 주요 의견만 간결하게 요약
- 찬반 의견이 있으면 함께 기록

**좋은 예**:
```markdown
* **왕택준**
  * ERD에 History 테이블 추가 제안
  * JWT 토큰 만료 시간 24시간으로 설정 제안

* **김경민**
  * Docker Compose 파일은 개발/운영 분리 필요
  * HTTPS 인증서는 Let's Encrypt 자동 갱신 설정
```

**작성 팁**:
- 누가 무엇을 제안했는지 명확히
- 기술적 근거가 있으면 간략히 포함
- 반대 의견도 기록

---

### 3. 결정 사항

**목적**: 회의에서 합의된 최종 결정을 명확히 기록

**작성 방법**:
- 결정 사항만 명확하게
- 조건이 있으면 함께 명시
- 누가 결정했는지보다 무엇이 결정되었는지 중심

**좋은 예**:
```markdown
* ERD에 History 테이블 추가 확정
* JWT 토큰 만료 시간 24시간으로 결정
* Docker는 개발/운영 환경 분리하여 작성
* Sprint 2는 김경민이 Scrum Master 담당
```

**작성 팁**:
- 결정의 근거는 주요 논의에 기록되어 있으므로 생략 가능
- 명확한 액션으로 이어질 수 있게 작성

---

### 4. 공통 작업

**목적**: 팀 전체가 함께 진행하거나 공유해야 할 작업 정리

**작성 방법**:
- 회의 중 / 회의 후로 구분
- 구체적인 액션 중심으로 작성
- 마감 기한이 있으면 명시

**좋은 예**:
```markdown
### 회의 중 공동 합의·진행 사항
* ERD Miro 보드에서 실시간 수정 완료
* 브랜치 전략 문서 함께 검토 완료

### 회의 후 공통 TODO
* API 명세서 Swagger로 작성 (10.08까지)
* 코드 리뷰 규칙 문서 검토 후 승인 (10.09까지)
```

---

### 5. 개별 담당

**목적**: 각 팀원의 개별 작업을 명확히 분배

**작성 방법**:
- 팀장, 팀원 가나다 순으로 정렬
- 역할/포지션 명시
- 구체적이고 측정 가능한 작업으로 작성
- 마감 기한 명시

**좋은 예**:
```markdown
* **왕택준 (PO/Tech Lead)**
  * ERD 최종 문서화 및 팀 공유 (10.08)
  * JWT 인증 API 구현 (10.10)
  * 시나리오 프롬프트 2개 작성 (10.12)

* **김경민 (SM/Fullstack Developer)**
  * Docker Compose 개발/운영 분리 작성 (10.09)
  * Nginx 설정 파일 작성 (10.10)
  * Hello World 배포 테스트 (10.11)
```

**작성 팁**:
- 작업이 너무 크면 세분화
- 의존성이 있는 작업은 순서 명시
- 예상 소요 시간이 길면 중간 체크포인트 설정

---

### 6. 리스크 및 이슈

**목적**: 현재 문제점과 잠재적 위험 요소 파악

**작성 방법**:
- 현재 발생한 문제 / 잠재적 리스크 구분
- 심각도 표시 (필요 시)
- 해결 방안이나 대응 계획 간략히 기록

**좋은 예**:
```markdown
* GPT-4o API 응답 속도 테스트 필요 (잠재적 성능 이슈)
* Janus 설치 시 포트 충돌 발생 → 포트 변경으로 해결
* Sprint 2 일정이 빡빡함 → 우선순위 재조정 필요
```

**작성 팁**:
- 문제만 나열하지 말고 대응 방향도 함께
- 해결된 이슈는 간략히 기록
- 미해결 이슈는 다음 회의 의제로 등록

---

### 7. 차기 회의 계획

**목적**: 다음 회의 일정과 다룰 내용 사전 공유

**작성 방법**:
- 구체적인 날짜와 시간
- 예상 의제는 3~5개 정도

**좋은 예**:
```markdown
* **일정**: 2025.10.08 14:00
* **예상 의제**:
  * Sprint 2 진행 상황 점검
  * API 명세서 검토
  * Docker 배포 테스트 결과 공유
```

## 작성 원칙

### 해야 할 것
- 회의 당일 또는 익일 내 작성 완료
- 객관적 사실 중심으로 기록
- 결정 사항과 담당 작업 명확히 구분
- 마감 기한 반드시 명시
- 간결하고 명확한 문장 사용

### 하지 말아야 할 것
- 감정적 표현 사용 금지
- 불필요한 장황한 설명 배제
- 애매모호한 표현 금지 ("나중에", "빨리", "대충")
- 개인 공격이나 비판적 표현 금지

## 작성 후 프로세스

### 1. 작성 완료 후
- docs 저장소의 `meeting-notes/` 폴더에 저장
- Discord 또는 Jira에 링크 공유
- 팀원 전원이 읽었는지 확인

### 2. 작업 분배
- 개별 담당 작업을 Jira 티켓으로 생성
- 담당자에게 할당 및 마감일 설정
- Sprint Backlog에 추가

### 3. 다음 회의 준비
- 차기 회의 일정을 팀 캘린더에 등록
- 예상 의제를 사전에 공유
- 미완료 작업은 차기 회의에서 점검
