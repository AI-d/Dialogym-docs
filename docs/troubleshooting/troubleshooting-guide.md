## 섹션별 작성 가이드

### 1. 문제 현상

**목적**: 발생한 문제를 명확하고 구체적으로 기술

**작성 방법**:
- 관찰 가능한 현상 중심으로 작성
- 에러 메시지는 정확히 복사
- 재현 조건을 구체적으로 명시

**좋은 예**:
```markdown
### 1-1. 주요 증상
* **문제**: WebRTC 연결이 로컬에서는 성공하지만 EC2에서 실패
* **증상**: ICE connection state가 'failed'로 전환
* **상황**: 서로 다른 네트워크에 있는 클라이언트 간 연결 시

### 1-2. 에러 정보
* **에러 메시지**: `ICE failed, add a STUN server and ensure it can be reached`
* **재현 조건**: 서로 다른 네트워크(외부 인터넷)에서 연결 시도
* **빈도**: 항상 발생

### 1-3. 환경 정보
* **운영체제**: Ubuntu 20.04 (EC2)
* **브라우저**: Chrome 120
* **관련 버전**: Janus 0.14.2
```

**작성 팁**:
- 스크린샷이나 로그 첨부
- 주관적 추측 배제
- 수치와 사실 중심

---

### 2. 원인 분석

**목적**: 문제의 근본 원인을 기술적으로 분석

**작성 방법**:
- 표면적 원인에서 근본 원인까지 단계적으로
- 코드나 설정의 구체적인 부분 명시
- 기술적 배경 지식 간략히 설명

**좋은 예**:
```markdown
### 2-1. 1차 분석
WebRTC는 P2P 연결을 위해 NAT traversal이 필요함

### 2-2. 2차 분석
Janus 설정에 STUN/TURN 서버 정보가 없어 외부 IP 발견 불가

### 2-3. 근본 원인
**문제점**:
- `janus.jcfg` 설정 파일에 STUN 서버 미설정
- ICE candidates에 외부 IP가 포함되지 않음
- NAT 환경에서 P2P 연결 불가능
```

---

### 3. 디버깅 과정

**목적**: 문제를 찾아낸 과정과 방법 공유

**작성 방법**:
- 실패한 시도도 포함
- 사용한 도구나 명령어 구체적으로
- 단계적 접근 과정 명시

**좋은 예**:
```markdown
### 3-1. 사용한 디버깅 기법
- Chrome DevTools WebRTC Internals
- Janus 로그 확인 (`/var/log/janus/`)
- Wireshark 패킷 분석

### 3-2. 핵심 문제 발견 과정
**1단계: 로컬 테스트**
- 같은 네트워크에서는 정상 작동 확인

**결과**: 네트워크 문제로 추정

**2단계: ICE Candidates 확인**
```javascript
pc.onicecandidate = (event) => {
  console.log('ICE candidate:', event.candidate);
};
```

**결과**: srflx (외부 IP) 후보가 생성되지 않음

**3단계: STUN 서버 확인**
- Janus 설정 파일에 STUN 서버 없음 발견
```

---

### 4. 해결 과정

**목적**: 실제 해결 방법과 구현 과정 기록

**작성 방법**:
- 실패한 해결책도 기록
- 최종 코드는 완전하게
- 왜 이 해결책을 선택했는지 근거 제시

**좋은 예**:
```markdown
### 4-1. 시도했던 해결책들

**A안: 공용 STUN 서버만 추가 (성공)**
```ini
[nat]
stun_server = stun.l.google.com
stun_port = 19302
```

**성공**: 외부 IP 발견 가능해짐

**B안: TURN 서버 추가 고려 (보류)**
- 비용과 복잡도 증가
- 현재는 STUN만으로 충분

### 4-2. 최종 해결책

Janus 설정 파일에 Google 공용 STUN 서버 추가

```ini
[nat]
stun_server = stun.l.google.com
stun_port = 19302
ice_lite = false
ice_tcp = false
```

**성공 이유**:
- 무료이고 안정적
- NAT traversal 성공률 높음
- 추가 인프라 불필요
```

---

### 5. 테스트 검증

**목적**: 해결책이 정말 효과적인지 입증

**작성 방법**:
- 정량적인 결과
- 여러 환경에서의 검증
- 회귀 테스트 결과

**좋은 예**:
```markdown
### 5-1. 로컬 환경 테스트
- 같은 네트워크: 정상 작동 확인

### 5-2. 프로덕션 환경 테스트
- 서로 다른 네트워크 10회 테스트: 10회 성공

### 5-3. 검증 결과
- **변경 전**: ICE connection state 'failed' (100%)
- **변경 후**: ICE connection state 'connected' (100%)
```

---

### 6. 성능 영향 분석

**목적**: 해결책이 시스템에 미치는 영향 평가

**작성 방법**:
- 구체적인 수치 제시
- 트레이드오프 명시

**좋은 예**:
```markdown
### 6-1. 변경 전후 비교
**측정 항목**:
- 연결 시간: 2초 → 3초 (+50%, STUN 조회 추가)
- 성공률: 20% → 100% (+400%)

### 6-2. 리소스 사용량
- **메모리**: 변화 없음
- **CPU**: 미미한 증가 (STUN 조회 시)
- **네트워크**: STUN 서버 통신 추가 (무시 가능)

### 6-3. 사용자 경험 영향
연결 시간이 1초 증가했지만 연결 성공률이 크게 향상되어 전체적으로 긍정적
```

---

### 7. 관련 이슈 및 예방책

**목적**: 유사한 문제 재발 방지

**작성 방법**:
- 구체적인 예방 액션
- 자동화 가능한 체크 항목

**좋은 예**:
```markdown
### 7-1. 유사한 함정 회피 방법
**위험한 패턴**:
- WebRTC 설정 시 STUN/TURN 서버 누락

**안전한 패턴**:
- 초기 설정부터 STUN 서버 필수 포함
- TURN 서버 백업 준비

### 7-2. 코드 리뷰 체크포인트
- [ ] Janus 설정에 STUN 서버 포함 여부
- [ ] ICE candidates 로그 확인
- [ ] 외부 네트워크 테스트 수행

### 7-3. 추가 예방 방법
- 배포 전 체크리스트에 WebRTC 연결 테스트 추가
- 모니터링 대시보드에 WebRTC 연결 성공률 추가
```

### 8. 결론 및 배운 점

**목적**: 문제 해결을 통해 얻은 인사이트 정리

**작성 방법**:
- 구체적인 성과 명시
- 다른 상황에 적용 가능한 지식
- 팀 전체에 도움이 되는 인사이트

**좋은 예**:
```markdown
### 8-1. 주요 성과
1. **WebRTC 연결 성공률 100% 달성**
2. **NAT traversal 메커니즘 이해**
3. **프로덕션 배포 전 체크리스트 개선**

### 8-2. 기술적 학습
**WebRTC NAT Traversal**:
- STUN으로 외부 IP 발견 가능
- TURN은 직접 연결 실패 시 릴레이 역할
- ICE framework가 자동으로 최적 경로 선택

**Janus 설정 관리**:
- 설정 파일 변경 시 재시작 필요
- 프로덕션 배포 전 필수 설정 체크리스트 작성 필요

### 8-3. 프로세스 개선
**배포 전 체크리스트에 추가**:
- [ ] WebRTC 설정 확인 (STUN/TURN)
- [ ] 외부 네트워크 테스트 수행
- [ ] ICE candidates 로그 확인

### 8-4. 장기적 개선 방향
**모니터링 강화**:
- WebRTC 연결 성공률 대시보드 추가
- ICE 연결 실패 알림 설정

**인프라 개선**:
- TURN 서버 구축 검토 (극단적 네트워크 환경 대응)
- 멀티 STUN 서버 설정 (백업)
```
