## 문서 개요 (Overview)

본 문서는 팀 내 기술 문서의 구조적 일관성과 품질을 유지하기 위해 작성되었습니다.

프로젝트가 성장하면서 문서의 형식이 제각각이 되면 검색성과 유지보수성이 저하됩니다.
이를 방지하기 위해 모든 문서가 따라야 하는 표준 템플릿을 정의하고, 작성 원칙과 승인 절차를 명확히 합니다.

본 템플릿은 Git 브랜치 전략, API 설계 문서, 배포 가이드 등 모든 기술 문서에 적용됩니다.

## Depth 규칙

문서의 제목 깊이를 통일하기 위한 규칙입니다.
고정 구조: `# 제목 → ## 파트 → ### 세부 → #### 부연`

| Depth | Markdown 문법 | 사용 목적 |
|-------|---------------|-----------|
| 1     | `#`           | 문서 제목 (유일해야 합니다) |
| 2     | `##`          | 주요 파트 구분 |
| 3     | `###`         | 세부 항목 |
| 4     | `####`        | 보조 설명 또는 하위 항목 |

## 문서 승인 절차 (Approval Workflow)

문서가 공식 상태(`Approved`)로 전환되기 위한 절차를 정의합니다.

- Draft: 초안 작성 중
- Approved: 검토자 승인 완료 후 공식 문서로 전환
- Deprecated: 더 이상 유효하지 않으며 archive/ 폴더로 이동

**기본 규칙:**

* 모든 문서는 `Draft` 상태로 시작합니다.
* `Reviewer` 검토 후 `Approved` 상태로 변경합니다.
* 승인 완료 시 `main` 브랜치에 병합합니다.
* 폐기된 문서는 `Deprecated` 상태로 전환하고 `archive/` 폴더로 이동합니다.

☑️ 모든 변경 사항은 반드시 **Pull Request(PR)**를 통해 승인 절차를 거쳐야 합니다.

## 문서 작성 원칙 (Authoring Rules)

문서의 품질과 일관성을 유지하기 위한 공통 규칙입니다.

| 항목 | 규칙 |
|------|------|
| 문체 | 모든 문서는 공식적이고 정중한 서술체로 작성합니다. (ex. ~합니다.) |
| 강조 | 불필요한 이모티콘은 사용하지 않습니다. |
| 예외 | 꼭 강조해야 하는 경우에만 ⚠️, ❌, ☑️ 등의 시각적 표시를 사용합니다. |
| 코드/명령어 | `code block`으로 표기합니다. |
| 형식 | 모든 문서는 Markdown(`.md`)으로 작성합니다. |
| 길이 | 불필요한 장문을 피하고 핵심 위주로 구성합니다. |
| 섹션 구분 | 각 `##` 파트 사이는 `---`로 구분합니다. |
| 파일명 | 파일명은 kebab-case로 `#` 제목을 영어로 키워드만 추출하여 작성합니다. (ex. `# Git 브랜치 전략 가이드` → `git-branch-strategy-guide.md`) |


그 외 `운영 원칙`, `예시`, `참고 자료` 등은 문서의 성격에 따라 자유롭게 추가합니다.
