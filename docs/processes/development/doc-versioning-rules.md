## 대상 독자 (Intended Audience)

* **팀원 전원**: 문서 상태(초안/확정본/수정본)를 확인하고 활용하는 담당자
* **팀 리더 / PM**: 문서 승인 및 버전 단계를 관리하는 책임자
* **운영자 / 외부 협력자**: 최신 문서 상태를 파악하고 참조하는 담당자
* **신규 합류자**: 문서 버전 체계를 이해하여 팀 문서 작성 및 활용 규칙에 빠르게 적응해야 하는 신규 멤버

---

## 핵심 요약 (Executive Summary)

본 문서는 팀에서 작성·관리하는 모든 산출물의 버전 관리 규칙을 정의합니다.
버전은 vX.Y 형식을 따르며, X는 주 버전(큰 변화), Y는 부 버전(작은 변경)을 의미합니다.
v0.x는 초안, v1.0은 최초 확정본, v1.x는 수정본, v2.0 이상은 대규모 개편을 나타냅니다.
모든 문서는 상단에 작성자, 문서 버전, 최종 수정일, 대상 독자를 명시하며, 파일명에는 버전을 포함하지 않습니다.
버전 변경 시 GitHub commit 메시지에 변경 사유를 명확히 기재하여 이력을 관리합니다.

## 문서 개요 (Overview)

본 문서는 팀 내 문서 버전 관리 규칙을 명확히 하기 위해 작성되었습니다.

프로젝트가 진행되면서 회의록, 기획서, 설계 문서, 보고서 등 다양한 산출물이 생성되고 수정됩니다.
일관된 버전 관리 규칙 없이 문서를 관리하면 최신 버전 파악이 어렵고, 문서의 신뢰성이 떨어집니다.

본 문서는 버전 표기 형식, 단계별 구분, 문서 내 기재 방법, GitHub 관리 규칙을 정의하여
팀 전체가 동일한 기준으로 문서를 작성하고 활용할 수 있도록 합니다.

## 버전 표기 형식

### 기본 형식

```
vX.Y
```

### 구성 요소

| 요소           | 의미                           | 예시                       |
|--------------|------------------------------|--------------------------|
| **X (주 버전)** | 문서의 큰 변화 반영 (초안 → 확정, 구조 변경) | v0.1 → v1.0, v1.5 → v2.0 |
| **Y (부 버전)** | 작은 변경 반영 (내용 추가/수정, 오탈자 교정)  | v1.0 → v1.1, v1.1 → v1.2 |

### 버전 예시

| 버전     | 의미                            |
|--------|-------------------------------|
| `v0.1` | 초안 (팀 검토 전)                   |
| `v1.0` | 최초 확정본                        |
| `v1.1` | 소규모 업데이트 (내용 추가, 오타 수정)       |
| `v1.2` | 추가 소규모 업데이트                   |
| `v2.0` | 문서 구조 대폭 개편 또는 새로운 프로젝트 기준 반영 |

## 버전 단계 구분

### v0.x (Draft/초안)

**상태**: 작성 중

**설명**: 작성자가 개인적으로 정리한 버전으로, 팀 전체 공유 전 단계입니다.

**특징**:

- 내용이 불완전하거나 변경 가능성이 높음
- 팀 검토 대기 중
- 참조용으로만 사용

**예시**: 기획서 초안, 회의록 임시 작성본

---

### v1.0 (Release/최초 확정본)

**상태**: 확정 완료

**설명**: 팀 전체 회의/합의를 거쳐 확정된 최초 문서 버전입니다.

**특징**:

- 프로젝트 공식 산출물로 활용 가능
- 팀 전체 승인 완료
- 기준 문서로 사용

**예시**: 최종 승인된 요구사항 명세서, 확정 ERD

---

### v1.x (Update/수정본)

**상태**: 부분 수정

**설명**: 기존 확정본에 대해 일부 수정이나 보강이 이루어진 버전입니다.

**특징**:

- 문서의 전체 구조에는 큰 변화 없음
- 내용 추가, 오타 수정, 표현 개선 등
- 이전 버전과 호환성 유지

**예시**: API 문서 엔드포인트 추가, 회의록 액션 아이템 업데이트

---

### v2.0 이상 (Major Update/대규모 개편)

**상태**: 대규모 변경

**설명**: 프로젝트 상황 변화에 따라 문서의 큰 틀이 바뀌었을 때 사용합니다.

**특징**:

- 목표 변경, 기능 구조 개편 등 전체 구조 변화
- 이전 버전과 호환성 낮음
- 새로운 프로젝트 단계 시작

**예시**: 프로젝트 방향 전환에 따른 기획서 재작성, 시스템 아키텍처 전면 개편

## GitHub 관리 규칙

### Commit 메시지 규칙

모든 문서는 commit 메시지에 버전 변경 사유를 명확히 기재합니다.

```bash
# 초안 작성
git commit -m "docs: 회의록 v0.1 초안 작성"

# 확정본 등록
git commit -m "docs: 기획서 v1.0 확정본 등록"

# 수정본 업데이트
git commit -m "docs: 설계 문서 v1.1 기능 설명 추가"

# 대규모 개편
git commit -m "docs: 요구사항 명세서 v2.0 프로젝트 방향 변경 반영"
```

### 파일명 규칙

문서 파일명에는 버전을 포함하지 않습니다.

```bash
# ☑️ 올바른 파일명
meeting-notes-2025-10-05.md
requirements-spec.md
design-document.md

# ❌ 잘못된 파일명
meeting-notes-v1.0-2025-10-05.md
requirements-spec-v2.0.md
design-document-v1.1.md
```

버전은 **문서 내부 메타 정보**로만 관리하며, 파일명에는 포함하지 않습니다.

### 버전 관리 이력

버전 관리 이력은 **GitHub commit 로그**를 기준으로 합니다.

```bash
# 커밋 로그 확인
git log --oneline -- docs/requirements-spec.md

# 특정 버전 시점 확인
git log --grep="v1.0" -- docs/

# 변경 내역 비교
git diff HEAD~1 docs/requirements-spec.md
```

## 버전 업데이트 시나리오

### 시나리오 1: 초안 작성

```markdown
상황: 회의록을 처음 작성
버전: v0.1
상태: Draft
Commit: "docs: 회의록 v0.1 초안 작성"
```

---

### 시나리오 2: 팀 검토 후 확정

```markdown
상황: 초안이 팀 검토를 거쳐 확정됨
버전: v0.1 → v1.0
상태: Draft → Approved
Commit: "docs: 회의록 v1.0 팀 검토 완료 및 확정"
변경 이력 추가: v1.0 | 2025.10.05 | 왕택준 | 팀 검토 완료 및 확정
```

---

### 시나리오 3: 소규모 수정

```markdown
상황: 오타 수정 및 액션 아이템 업데이트
버전: v1.0 → v1.1
상태: Approved (유지)
Commit: "docs: 회의록 v1.1 오타 수정 및 액션 아이템 업데이트"
변경 이력 추가: v1.1 | 2025.10.06 | 왕택준 | 오타 수정 및 액션 아이템 업데이트
```

---

### 시나리오 4: 대규모 개편

```markdown
상황: 프로젝트 방향 변경으로 요구사항 전면 재작성
버전: v1.5 → v2.0
상태: Approved (유지)
Commit: "docs: 요구사항 명세서 v2.0 프로젝트 방향 변경 반영"
변경 이력 추가: v2.0 | 2025.11.01 | 왕택준 | 프로젝트 방향 변경에 따른 전면 재작성
```

---

### 시나리오 5: 문서 폐기

```markdown
상황: 더 이상 사용하지 않는 문서
버전: v1.3 (유지)
상태: Approved → Deprecated
Commit: "docs: 구 설계 문서 v1.3 Deprecated 처리"
변경 이력 추가: v1.3 | 2025.12.01 | 왕택준 | 신규 설계로 대체되어 폐기
파일 이동: docs/ → archive/
```
