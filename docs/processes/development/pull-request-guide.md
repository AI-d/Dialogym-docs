## 핵심 요약 (Executive Summary)

본 문서는 프로젝트 팀에서 사용하는 PR 작성 규칙과 Jira 연동 방법을 정의합니다.
PR 제목은 TRAIN-이슈번호 타입: 설명 형식을 따르며, 본문은 작업 내용 요약, 변경 사항, 테스트 방법을 포함합니다.
PR 제목에 TRAIN-XX를 포함하면 Jira 이슈와 자동 연동되며, 병합 시 이슈 상태가 자동으로 업데이트됩니다.
PR 크기는 100-300줄을 권장하며, 1000줄 이상은 분할이 필요합니다.
최소 1명의 Approve와 CI 통과 후 Squash & Merge로 병합합니다.

## 문서 개요 (Overview)

본 문서는 PR 작성 규칙과 프로세스를 명확히 하기 위해 작성되었습니다.

PR은 코드 변경을 검토하고 병합하는 핵심 프로세스입니다. 명확한 규칙 없이 작성되면 리뷰가 지연되고, 변경 내용을 이해하기 어려우며, 작업 추적이 불가능해집니다.

본 문서는 PR 제목 형식, 본문 템플릿, Jira 연동 방법, 적정 크기, 체크리스트를 포함하여 효율적인 PR 프로세스를 지원합니다.

## PR 작성 규칙

### 제목 형식

```
TRAIN-이슈번호 타입: PR 설명
```

### 제목 예시

```bash
# 기능 추가
TRAIN-12 feat: 사용자 인증 시스템 구현

# 버그 수정
TRAIN-23 fix: 로그인 validation 오류 해결

# 리팩토링
TRAIN-34 refactor: 유저 서비스 아키텍처 개선

# 성능 개선
TRAIN-45 perf: DB 쿼리 최적화
```

### 본문 작성 원칙

다음 내용을 포함합니다.

1. **작업 내용 요약**: 무엇을 변경했는지
2. **변경 사항**: 주요 변경된 파일/모듈
3. **테스트 방법**: 리뷰어가 확인할 방법
4. **Breaking Changes**: API 변경 시 명시
5. **스크린샷**: UI 변경 시 첨부

```markdown

<!-- 리뷰어가 이 기능을 테스트할 수 있는 방법 -->

2. Google 로그인 버튼 클릭
3. 인증 후 대시보드 리다이렉션 확인

<!-- Before/After 스크린샷 첨부 -->

## Breaking Changes

<!-- API 변경, DB 마이그레이션 등이 필요한 경우 명시 -->

- [ ] 데이터베이스 마이그레이션 필요
- [ ] 환경 변수 추가 필요
- [ ] API 스펙 변경

## 체크리스트

- [ ] 로컬에서 기능 테스트 완료
- [ ] 기존 테스트 통과 확인
- [ ] 린트 검사 통과
- [ ] 문서 업데이트 (필요 시)
- [ ] Breaking Changes 명시 (해당 시)

## 관련 Jira 이슈
```

## Jira 연동

### 자동 연동

PR 제목에 TRAIN-XX 포함 시 다음이 자동으로 수행됩니다.

- Jira 이슈 "Development" 탭에 PR 자동 연결
- PR 상태가 Jira에 실시간 반영
- PR 머지 시 Jira 이슈 자동 업데이트

### PR 본문에 Jira 링크 추가

```markdown
## 관련 Jira 이슈

https://your-team.atlassian.net/browse/TRAIN-12

또는

TRAIN-12
```

### 여러 이슈 참조

```markdown
## 관련 Jira 이슈

TRAIN-12 (메인 이슈)
TRAIN-23 (관련 이슈)
TRAIN-34 (참조)
```

## PR 크기 가이드

### 권장 크기

| 크기          | 변경 라인 수     | 권장 사항   |
|-------------|-------------|---------|
| **Small**   | 100-300 라인  | 권장 ☑️   |
| **Medium**  | 300-500 라인  | 허용      |
| **Large**   | 500-1000 라인 | 최대 한도   |
| **X-Large** | 1000+ 라인    | 분할 필요 ❌ |

### 대규모 변경 분할 전략

```bash
# 나쁜 예: 하나의 거대한 PR
TRAIN-100 feat: 전체 사용자 관리 시스템 구현

# 좋은 예: 여러 작은 PR로 분할
TRAIN-101 feat: 사용자 모델 및 DB 스키마
TRAIN-102 feat: 사용자 인증 API
TRAIN-103 feat: 사용자 프로필 UI
TRAIN-104 feat: 사용자 관리 대시보드 통합
```

## 체크리스트

### PR 생성 전

다음 사항을 체크합니다.

- dev 브랜치 최신화 완료
- 로컬 테스트 통과
- 린트 검사 통과
- 커밋 메시지 규칙 준수
- 브랜치명 규칙 준수

### PR 생성 시

다음 사항을 체크합니다.

- 제목 형식 준수 (TRAIN-XX type: description)
- 본문 템플릿 작성
- 테스트 방법 명시
- 스크린샷 첨부 (UI 변경 시)
- Breaking Changes 명시 (해당 시)
- 리뷰어 지정
- 라벨 추가 (frontend, backend 등)

### PR 승인 후

다음 사항을 체크합니다.

- CI 통과 확인
- 리뷰어 Approve 1명 이상
- 충돌 해결 완료
- Squash & Merge 수행
- 브랜치 삭제 (자동/수동)
- Jira 이슈 상태 확인

## PR 상태별 라벨

### GitHub 라벨 활용

**상태**:

- WIP: 작업 중 (리뷰 요청 전)
- Ready for Review: 리뷰 준비 완료
- Changes Requested: 수정 요청됨
- Approved: 승인됨

**영역**:

- frontend: 프론트엔드
- backend: 백엔드
- database: 데이터베이스
- docs: 문서

**우선순위**:

- priority: high: 높음
- priority: medium: 보통
- priority: low: 낮음

## Draft PR 활용

### Draft PR 사용 시점

다음 상황에서 Draft PR을 사용합니다.

1. **조기 피드백**: 설계 단계에서 의견 수렴
2. **WIP 공유**: 작업 진행 상황 공유
3. **실험적 변경**: 아이디어 검증

### Draft → Ready 전환 시점

다음 조건을 충족할 때 전환합니다.

- 모든 기능 구현 완료
- 테스트 통과
- 문서 업데이트 완료
- 자체 리뷰 완료

## PR 병합 절차

### 1. 리뷰 요청

```bash
1. PR 생성
2. 리뷰어 지정 (최소 1명)
3. Discord/Slack에 리뷰 요청 알림
```

### 2. 리뷰 진행

```bash
1. 리뷰어가 코드 검토
2. 피드백 제공
3. 수정 요청 또는 승인
```

### 3. 수정 반영

```bash
1. 피드백에 따라 수정
2. 추가 커밋 푸시
3. 재검토 요청
```

### 4. 병합

```bash
1. Approve 1명 이상 확보
2. CI 통과 확인
3. Squash & Merge 실행
4. 브랜치 삭제
```
