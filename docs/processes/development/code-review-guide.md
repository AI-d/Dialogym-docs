## 핵심 요약 (Executive Summary)

본 문서는 프로젝트 팀에서 사용하는 코드 리뷰 규칙과 절차를 정의합니다.
모든 PR은 최소 1명의 Approve를 받아야 하며, 자기 PR은 본인이 병합할 수 없습니다.
리뷰 요청 후 24시간 내에 피드백을 제공해야 하며, 기능 테스트는 로컬에서 직접 실행하여 확인합니다.
피드백은 Must Fix, Should Fix, Consider, Nitpick 카테고리로 구분하여 우선순위를 명확히 합니다.
작성자는 방어적이지 않고 열린 마음으로 피드백을 수용하며, 리뷰어는 구체적인 위치와 개선 방안을 제시합니다.

## 문서 개요 (Overview)

본 문서는 팀 내 코드 리뷰 규칙을 명확히 하기 위해 작성되었습니다.

코드 리뷰는 코드 품질을 유지하고, 버그를 조기에 발견하며, 팀원 간 지식을 공유하는 중요한 프로세스입니다. 그러나 명확한 규칙 없이 진행되면 리뷰가 지연되거나 피드백의 질이 떨어질 수 있습니다.

본 문서는 리뷰어와 작성자의 역할, 피드백 작성 방법, 우선순위 구분, 체크리스트를 포함하여 효율적이고 건설적인 코드 리뷰 문화를 구축합니다.

## 기본 원칙

### 리뷰 규칙

다음 규칙을 준수합니다.

1. **최소 1명 Approve 필수**: PR 병합 전 필수
2. **자기 PR 병합 금지**: 본인이 올린 PR은 타인이 병합
3. **24시간 내 리뷰**: 리뷰 요청 후 1일 내 피드백
4. **기능 테스트 필수**: 로컬에서 직접 실행 확인

### 리뷰 우선순위

| 우선순위         | 상황            | 응답 시간  |
|--------------|---------------|--------|
| **Critical** | 보안, 데이터 손실 위험 | 즉시     |
| **High**     | 핵심 기능, API 변경 | 4시간 이내 |
| **Medium**   | 일반 기능, UI 개선  | 1일 이내  |
| **Low**      | 문서, 스타일 변경    | 2일 이내  |

## 리뷰어 가이드

### 리뷰 절차

#### 1단계: PR 개요 파악

다음 사항을 체크합니다.

- PR 제목과 설명 확인
- Jira 이슈 내용 확인
- 변경 파일 목록 확인
- PR 크기 적정성 판단

#### 2단계: 코드 검토

**코드 품질**:

- 네이밍 명확성
- 함수/클래스 크기 적정성
- 주석 적절성
- 코딩 스타일 일관성

**기능 검증**:

- 요구사항 충족도
- 엣지 케이스 처리
- 에러 핸들링
- 성능 영향도

**보안 및 안정성**:

- 보안 취약점 여부
- 데이터 검증 로직
- 권한 체크
- 리소스 누수 가능성

#### 3단계: 로컬 테스트

```bash
# PR 브랜치 가져오기
git fetch origin
git switch feature/TRAIN-12-description

# 의존성 설치 및 빌드
npm ci
npm run build

# 테스트 실행
npm test
npm run e2e

# 기능 수동 테스트
npm run dev
# 브라우저에서 기능 확인
```

#### 4단계: 피드백 제공

피드백 작성 시 다음을 준수합니다.

1. 구체적인 위치 지적
2. 문제점과 이유 설명
3. 개선 방안 제안
4. 칭찬도 함께 제공

## 작성자 가이드

### PR 생성 전

다음 사항을 셀프 체크합니다.

- 로컬 테스트 통과
- 린트 검사 통과
- 커밋 메시지 규칙 준수
- PR 크기 적정성 확인
- 관련 문서 업데이트

### 피드백 받을 때

**태도**:

- 방어적이지 않고 열린 마음으로
- 피드백을 학습 기회로 활용
- 이해 안 되면 질문하기

**대응**:

1. 피드백 내용 이해
2. 동의하면 즉시 수정
3. 불명확하면 질문
4. 의견 차이는 논의

### 수정 반영

```bash
# 피드백 반영 커밋
git add .
git commit -m "TRAIN-12 review: 에러 처리 로직 개선"
git push origin feature/TRAIN-12-description

# 리뷰어에게 알림
"피드백 반영 완료했습니다. 재검토 부탁드립니다."
```

## 피드백 가이드라인

### 피드백 카테고리

| 카테고리           | 의미        | 대응       |
|----------------|-----------|----------|
| **Must Fix**   | 반드시 수정 필요 | 수정 후 재검토 |
| **Should Fix** | 수정 권장     | 논의 가능    |
| **Consider**   | 검토 제안     | 선택 사항    |
| **Nitpick**    | 사소한 스타일   | 선택 사항    |

### 좋은 피드백 예시

#### Must Fix

```markdown
**Must Fix**: SQL 인젝션 취약점

현재 코드:

```sql
query = "SELECT * FROM users WHERE id = '" + userId + "'";
```

수정 제안:

```sql
query
= "SELECT * FROM users WHERE id = ?";
stmt.setString
(1, userId);
```

이유: 파라미터 바인딩을 사용하지 않으면 SQL 인젝션 공격에 취약합니다.

```

#### Should Fix

```markdown
**Should Fix**: 에러 처리 누락

현재 코드:
```js
async function fetchUser(id) {
  const response = await fetch(`/api/users/${id}`);
  return response.json();
}
```

개선 제안:

```js
async function fetchUser(id) {
    try {
        const response = await fetch(`/api/users/${id}`);
        if (!response.ok) throw new Error('Failed to fetch user');
        return response.json();
    } catch (error) {
        console.error('Error fetching user:', error);
        throw error;
    }
}
```

이유: 네트워크 오류 처리가 없어 사용자에게 적절한 피드백을 제공할 수 없습니다.

```

#### Consider

```markdown
**Consider**: 함수 분리

현재 `processOrder` 함수가 100줄이 넘습니다.
다음과 같이 분리하는 것을 고려해보세요:
- `validateOrder`
- `calculateTotal`
- `applyDiscount`
- `saveOrder`

장점: 테스트와 유지보수가 쉬워집니다.
```

#### Nitpick

```markdown
**Nitpick**: 변수명

`data`보다는 `userProfile`이 더 명확할 것 같습니다.
```

## 체크리스트

### 리뷰어 체크리스트

**코드 품질**:

- 네이밍이 명확한가
- 함수가 적절한 크기인가
- 주석이 필요한 곳에 있는가
- 코딩 스타일이 일관적인가

**기능**:

- 요구사항을 충족하는가
- 엣지 케이스를 처리하는가
- 에러 핸들링이 적절한가

**보안**:

- SQL 인젝션 취약점은 없는가
- XSS 취약점은 없는가
- 인증/권한 체크가 있는가

**성능**:

- 불필요한 연산이 없는가
- 메모리 누수 가능성은 없는가
- DB 쿼리가 최적화되었는가

**테스트**:

- 로컬에서 기능이 작동하는가
- 기존 테스트가 통과하는가
- 새 테스트가 필요한가

### 작성자 체크리스트

**PR 생성 전**:

- 로컬 테스트 통과
- 린트 검사 통과
- 커밋 메시지 규칙 준수
- dev 최신화 완료

**PR 생성 시**:

- 제목 형식 준수
- 본문 템플릿 작성
- 테스트 방법 명시
- 스크린샷 첨부 (UI 변경 시)
- Breaking Changes 명시 (해당 시)

**피드백 받은 후**:

- 모든 피드백 확인
- Must Fix 항목 수정
- Should Fix 항목 논의/수정
- 재검토 요청

## 리뷰 시간 단축 팁

### 작성자 측

다음 방법으로 리뷰 시간을 단축합니다.

1. **작은 PR**: 300줄 이하로 유지
2. **명확한 설명**: PR 본문 상세 작성
3. **자체 리뷰**: 생성 전 스스로 검토
4. **Draft PR**: 조기 피드백 활용

### 리뷰어 측

다음 방법으로 리뷰 시간을 단축합니다.

1. **빠른 응답**: 24시간 내 피드백
2. **구체적 피드백**: 위치와 이유 명시
3. **중요도 표시**: Must/Should/Consider 구분
4. **칭찬도 함께**: 잘한 부분 언급
