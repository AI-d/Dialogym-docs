## 핵심 요약 (Executive Summary)

본 문서는 프로젝트 팀에서 사용하는 Discord 채널 구조와 운영 원칙을 정의합니다.
Discord는 실시간 소통 도구로, 7개 채널(텍스트 6개, 음성 1개)로 구성됩니다.
공지사항은 스프린트 시작/종료와 긴급 상황을 알리며, 데일리는 스탠드업 기록 전용입니다.
개발 질문은 5분 내 답변 불가능하면 Discussions로 이동하며, 잡담은 비업무 관련 소통 공간입니다.
채널 최소화 원칙을 유지하며, 필요 시에만 새 채널을 추가합니다.

## 문서 개요 (Overview)

본 문서는 Discord 채널 구조와 운영 원칙을 명확히 하기 위해 작성되었습니다.

팀 소통이 분산되면 중요한 정보를 놓치거나 잘못된 채널에서 대화하여 혼란이 발생합니다. 이를 방지하기 위해 채널별 목적과 사용 규칙을 명확히 정의합니다.

본 문서는 7개 Discord 채널의 역할, 사용 예시, 운영 원칙을 포함하여 효율적인 실시간 소통을 지원합니다.

---

## 채널 구조 개요

현재 Discord 서버는 7개 채널로 구성됩니다.

- **텍스트 채널(6개)**: 공지사항, 일반, daily, backend, frontend, 잡담
- **음성 채널(1개)**: 회의실

---

## 텍스트 채널

### 1. 공지사항

**목적**: 팀 전체 공지 전달 채널

**사용 예시**:
- 스프린트 시작/종료
- 마일스톤
- 배포 완료 등 주요 이벤트 기록
- 긴급 상황(서버 다운, 배포 실패 등)은 @everyone 멘션 허용

---

### 2. 일반

**목적**: 기본 업무 대화 채널

**사용 예시**:
- 작업 공유
- 질문
- 논의 등 대부분의 소통 진행
- 단순 질의응답은 즉시 해결
- 긴 논의는 Jira/Discussions로 이동

---

### 3. daily

**목적**: 데일리 스탠드업 기록 전용 채널

**작성 원칙**: 매일 아침 작성
- 오늘 할 일
- 어제 한 일
- 블로킹 이슈

**비고**: Jira Sprint와 병행하며, 요약 기록을 보관합니다.

---

### 4. backend

**목적**: 백엔드 개발 전용 채널

**사용 예시**:
- API 설계
- DB 쿼리
- 서버 로직
- 배포 이슈 공유
- CI/CD, 서버 인프라 관련 논의 포함

---

### 5. frontend

**목적**: 프론트엔드 개발 전용 채널

**사용 예시**:
- UI/UX 논의
- React 컴포넌트 개발
- 상태 관리
- 디자인 및 화면 설계 피드백 포함

---

### 6. 잡담

**목적**: 자유로운 대화 공간

**사용 예시**:
- 비업무 관련 소통 (취미, 일상, 친목)
- 단, 업무 집중 시간에는 절제

---

## 음성 채널

### 7. 회의실

**목적**: 음성 회의 전용 채널

**사용 예시**:
- 스프린트 플래닝
- 회고
- 리뷰
- 정기 회의 시 사용
- 필요 시 화면 공유 활용 가능

---

## 운영 원칙

### 1. 채널 최소화 원칙

다음 원칙을 준수합니다.

- 불필요한 세분화 금지
- 필요 시 확장 (예: 인프라 전담 인원 합류 시 infra 채널 추가)

### 2. 긴급성 기준 사용

다음과 같이 긴급성 기준을 적용합니다.

- 공지사항: 필수 공지/긴급 알림
- 일반: 대부분의 업무 대화
- backend/frontend: 기술적 논의

### 3. 비업무/업무 구분

다음과 같이 구분합니다.

- 잡담은 자유롭게 사용
- 단, 업무 채널은 업무 외 대화 자제

### 4. Daily 기록 일관성

다음 형식을 준수합니다.

- "오늘 / 어제 / 블로킹" 세 가지 항목 필수
- 동일 포맷 유지
