## 문서 개요

본 문서는 프로젝트의 역할 구조를 명확히 정의하여 팀 내 책임과 권한의 범위를 구체화하고, 균형 잡힌 협업 체계를 확립하기 위해 작성되었습니다. 애자일 스크럼 방법론을 기반으로 하며, **PO 1명 + 공동 SM
2명** 구조를 통해 견제와 균형을 유지합니다.

## 프로젝트 개요

| 항목      | 내용                     |
|---------|------------------------|
| 프로젝트 기간 | 5주 (Sprint 1주 단위 × 5회) |
| 운영 방식   | Agile Scrum            |
| 팀 규모    | 3명 (PO 1명 + 공동 SM 2명)  |

## 의사결정 구조

| 영역          | 주체            | 방식                            |
|-------------|---------------|-------------------------------|
| 제품 방향(What) | Product Owner | 비전·우선순위 제시 → 팀 협의             |
| 기술/설계(How)  | 전체 팀          | 토론 후 합의, 불가 시 Tech Lead 최종 판단 |
| 일정 수립       | 공동 SM         | 팀 합의 기반 예측·조율                 |
| 품질 승인       | 팀 전체          | 교차 테스트·리뷰 후 PO 최종 승인          |

## 품질 관리 (QA)

* **운영 원칙**: 개발자 자체 테스트 + 상호 교차 검증(전원 공동 책임)
* **절차**:

    1. 기능 단위 테스트 작성 → 2) PR 시 리뷰·기능 확인 → 3) 배포 전 통합 테스트 → 4) PO 최종 승인

## 운영 철학

초기에는 PO가 협업 구조를 구축하고, 이후 스프린트별로 SM을 순환 배정하여 **모든 구성원이 리더십을 경험**합니다. 기술 의사결정은 민주적 협의를 원칙으로 하며, Tech Lead가 방향을 제시하되 최종 결정은
팀 합의에 기반합니다. **문서 기반 협업**과 **투명한 프로세스**로 품질·속도·팀워크의 균형을 지향합니다.
